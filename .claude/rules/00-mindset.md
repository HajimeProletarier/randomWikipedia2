# 開発マインドセット総則

すべてのルールに通底する、根本的な考え方。

## コードは負債である

- 書いたコードは全て保守コスト
- 最良のコードは「書かなかったコード」
- 削除できるコードは最高のコード

## シンプルさは力

> "Simplicity is the ultimate sophistication." - Leonardo da Vinci

- 複雑さは徐々に忍び込む
- 常に「これ、もっとシンプルにできないか？」と問う
- YAGNI (You Aren't Gonna Need It) を忘れない

## 変更に強い設計

1. **疎結合**: コンポーネント間の依存を最小に
2. **高凝集**: 関連する機能は近くに
3. **抽象化**: 変わりやすい部分を隠蔽

## フィードバックループを短く

- 長いビルド時間は生産性を殺す
- テストは速く、頻繁に実行
- CI/CD で継続的にフィードバック

## 問題の根本を解決する

```
症状: ユーザーからのバグ報告
表面的対処: その場しのぎのif文
根本解決: なぜそのバグが入り込めたのかを分析し、仕組みで防ぐ
```

## 名前は大事

- 名前は最高のドキュメント
- 良い名前が思いつかないなら、設計が悪い可能性
- 名前を変えることを恐れない

## コミュニケーションとしてのコード

- コードは人間が読むものである
- 書く時間より読む時間の方が長い
- 未来の自分は他人だと思え

## 完璧主義の罠

- 80%の完成度で価値を届ける方が、100%を目指して何も届けないより良い
- ただし「動けばいい」と「十分に良い」は違う
- 技術的負債は認識した上で、意図的に負う

## 継続的な改善

- 昨日の自分より今日の自分
- コードベースは生き物、常に進化する
- ボーイスカウトルール: 来たときより美しく
